# Charter v0.1 Quality Gates

## Purpose

This document defines the required quality gates for v0.1. The gates are focused
on local requirements, drafts, immutable versions, acceptance criteria,
manual/proposed trace links, JSON CLI envelopes, and state/contract tests.

## Required Gates

| Gate | Command | Required Result | Scope |
|---|---|---|---|
| Formatting and lint | `make lint` | Exit 0 | Source and tests |
| Static typing | `make typecheck` | Exit 0 | Strict mypy |
| Unit tests | `make test` | Exit 0 | All non-network tests |
| Coverage/CI bundle | `make ci` | Exit 0 | Full local gate |
| Contract tests | `uv run pytest tests/contracts -q` | Exit 0 | JSON fixtures and envelopes |
| State-machine tests | `uv run pytest tests/state -q` | Exit 0 | Requirement and trace lifecycles |

`make ci` remains the required final local gate. Contract and state-machine
tests must be included in the default pytest suite once implemented, so `make ci`
is sufficient for normal verification.

## Unit Test Requirements

Unit tests must cover:

- requirement ID generation and duplicate detection;
- draft creation and update;
- approval creates immutable version `1`;
- supersede creates version `n+1`;
- deprecate preserves approved version content;
- acceptance criterion creation and lookup;
- trace link proposal;
- trace link acceptance and rejection;
- JSON success and error envelope helpers;
- CLI `--json` output for v0.1 commands.

## State-Machine Test Requirements

Requirement lifecycle tests:

- `draft -> approved`
- `draft -> rejected`
- `approved -> draft_revision -> approved`
- `approved -> deprecated`
- approved version text cannot be mutated;
- stale `expected_version` returns `CONFLICT`.

Trace lifecycle tests:

- `proposed -> accepted`
- `proposed -> rejected`
- `accepted -> stale`
- `accepted -> orphaned`
- inverted canonical relation returns `VALIDATION`;
- high/security/safety code link remains `proposed` unless accepted.

## Contract Test Requirements

Contract tests must validate fixtures for:

- success envelope;
- error envelope;
- list envelope;
- batch envelope;
- requirement version;
- trace link proposed/accepted/stale/orphaned;
- MCP side-effect metadata;
- CLI JSON output.

Validation checks:

- required keys present;
- enum closure enforced;
- `schema` values versioned;
- `ok` matches success/error shape;
- warnings always present;
- mutating tool metadata includes actor/idempotency/dry-run policy.

## Review Gates

Before v0.1 implementation starts:

- ADR-001 through ADR-004 are accepted.
- v0.1 implementation plan is approved.
- v0.1 work package execution guide is reviewed.
- RTM is reviewed for v0.1 scope.
- Contract fixture plan is approved.

Before v0.1 merge:

- `make ci` passes.
- Contract fixtures exist for every v0.1 JSON command.
- No v0.2+ scope has been implemented without explicit approval.
- No federation integration code exists except inert contracts/fixtures.

## Non-Goals For v0.1 Tests

Do not require:

- live Clarion;
- live Filigree;
- live Wardline;
- live Legis;
- MCP mutation execution;
- verification record freshness;
- baselines;
- impact engine.

Those belong to later slices.
