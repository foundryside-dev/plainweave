## Architecture Slice

The v0.1 implementation should use this layering:

```text
charter.cli
  -> charter.cli_commands
    -> charter.service
      -> charter.store
        -> sqlite3

charter.models, charter.errors, charter.envelopes, and charter.paths are
shared by the layers above them. No peer-tool integration layer exists in v0.1.
```

Layer rules:

- `cli.py` owns parser construction and process exit behavior only.
- `cli_commands.py` converts argparse namespaces to service calls and envelopes.
- `service.py` owns domain state transitions, actor/idempotency checks, and events.
- `store.py` owns SQL, migrations, transactions, and row mapping.
- `models.py` owns typed dataclasses and closed enums.
- `envelopes.py` owns JSON envelope shape and producer metadata.
- `errors.py` owns closed error codes and exceptions.
- `paths.py` owns project discovery and `.charter/` paths.

## Cross-Cutting Contracts

### Project Identity

`charter init` creates `.charter/charter.db` and records a project key. Tests
should pass `--project-key AUTH` for deterministic IDs. If omitted, v0.1 may use
`LOCAL` or a sanitized directory name, but the choice must be documented in
`charter doctor --json`.

Stable requirement IDs:

```text
display_id: REQ-<PROJECT>-<0001>
stable_id:  charter:req:<PROJECT>:<0001>
```

The database may store both `display_id` and `stable_id`. Agent-facing JSON must
include `id`, `stable_id`, and `version` for approved versions.

### Actors And Idempotency

Mutations require `actor`. Approval-style mutations also require an
`idempotency_key` or an implementation-generated deterministic key derived from
the command, actor, target, and expected version.

Minimum idempotency behavior:

- repeated `init` returns success with `created: false`;
- repeated approval with the same idempotency key returns the original approved version;
- conflicting approval with the same target but different expected version returns `CONFLICT`;
- mutating commands without actor return `VALIDATION`.

### Error Codes And Exit Codes

Use the ADR-004 closed error-code set:

```text
VALIDATION, NOT_FOUND, CONFLICT, POLICY_REQUIRED, PEER_ABSENT,
PEER_STALE, PEER_CONTRACT, LOCKED, UNSUPPORTED, INTERNAL
```

Exit policy:

- `0`: success, including advisory warnings;
- `1`: local advisory failure only when `--fail-on` requests it;
- `2`: input/config/protocol error;
- `3`: required peer unavailable/stale;
- `4`: internal error.

No v0.1 command should require a peer, so exit `3` is contract-covered but not
normally emitted by local-core behavior.

### Requirement Lifecycle

State transitions:

```text
draft -> approved
draft -> rejected
approved -> draft_revision -> approved
approved -> deprecated
approved -> superseded
```

Implementation stance:

- `requirements` stores stable identity and pointers only.
- `requirement_drafts` stores mutable draft text.
- `requirement_versions` stores immutable approved text.
- `requirement_versions.statement_hash` is computed from canonical statement text.
- Superseding an approved requirement creates a new approved version and marks the previous version superseded.
- Deprecating a requirement changes lifecycle status without mutating approved version text.

### Acceptance Criteria

Criteria are version-contextual. In v0.1, adding criteria should target an active
draft. On approval, criteria are associated with the approved version. Adding
criteria to an already-approved requirement without an active draft returns
`POLICY_REQUIRED` with a hint to supersede/revise the requirement.

Required fields:

```text
criterion_id, requirement_id, draft_id?, version?, position, text, status,
created_by, created_at
```

### Trace-Link Ontology

Use ADR-003 canonical relation direction. v0.1 may store peer node kinds as
opaque references but must not refresh, verify, or mutate peer-owned facts.

Canonical examples:

```text
clarion_entity --satisfies--> requirement_version
file_ref --fragile_satisfies--> requirement_version
verification_method --verifies--> requirement_version
verification_record --evidences--> verification_method
test_selector --provides_evidence_for--> verification_method
filigree_issue --implements_work_for--> requirement_version
filigree_issue --resolves_gap--> gap
wardline_finding --violates--> acceptance_criterion
legis_attestation --attests--> requirement_version
```

Trace states:

```text
proposed -> accepted
proposed -> rejected
proposed -> stale
accepted -> stale
accepted -> orphaned
stale -> proposed
stale -> accepted
stale -> rejected
orphaned -> proposed
orphaned -> rejected
```

Accepted traceability must remain distinguishable from proposed, inferred,
imported, stale, and orphaned facts in both storage and JSON output.

## v0.1 CLI Contract

All agent-facing commands support `--json`. Human output can stay concise and is
not a contract.

```text
charter init [--project-key AUTH] [--json]
charter doctor [--json]

charter req add --title TITLE --statement TEXT --actor ACTOR [--json]
charter req edit REQ_ID --statement TEXT [--title TITLE] --actor ACTOR [--json]
charter req show REQ_ID [--json]
charter req search [QUERY] [--json]
charter req approve REQ_ID --actor ACTOR --expected-version N [--idempotency-key KEY] [--json]
charter req supersede REQ_ID --title TITLE --statement TEXT --actor ACTOR --expected-version N [--idempotency-key KEY] [--json]
charter req deprecate REQ_ID --actor ACTOR --expected-version N [--idempotency-key KEY] [--json]

charter criterion add REQ_ID --text TEXT --actor ACTOR [--json]
charter criterion list REQ_ID [--json]

charter trace propose --from-kind KIND --from-id ID --relation REL --to-kind KIND --to-id ID --actor ACTOR [--confidence N] [--json]
charter trace accept LINK_ID --actor ACTOR [--json]
charter trace reject LINK_ID --actor ACTOR --reason TEXT [--json]
charter trace list [--requirement REQ_ID] [--state STATE] [--json]
```

Out of scope for v0.1 CLI:

- verification records;
- baselines;
- impact reports;
- MCP server commands beyond help text;
- live Clarion, Filigree, Wardline, or Legis calls;
- import/export;
- generated session context.
